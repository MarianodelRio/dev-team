# dev-team — CLAUDE.md

## What is dev-team?

**dev-team** is a framework for building software with a team of AI agents via parallel spec-driven development. Starting from a vague idea or an existing codebase, it guides you through design and planning, then executes the implementation using specialized agents working in parallel — each owning their module, each enforcing quality.

Single source of truth: git. No external services, no databases, no dashboards.

---

## How it works

```
IDEA.md  →  /bootstrap  →  design.md + plan.md + tasks/  →  /orchestrate  →  /prepare-pr  →  /done
```

1. **Write your idea** in `IDEA.md` (as vague as you want)
2. **Run `/bootstrap`** — conversational design session, generates architecture + tasks + infra
3. **Run `/orchestrate`** — agents pick tasks, implement, push PRs
4. **Run `/prepare-pr T-XXX`** — opens the PR with full review
5. **Merge on GitHub, run `/done T-XXX`** — marks done, reports what's unblocked

---

## Folder structure

```
IDEA.md                    ← your project idea (fill this first)
CLAUDE.md                  ← this file — rules and architecture
devteam.config.yml         ← framework configuration
design.md                  ← architecture, modules, contracts (generated by /bootstrap)
plan.md                    ← phases and dependency graph (generated by /bootstrap)

.claude/
  commands/                ← skills (/bootstrap, /orchestrate, /prepare-pr, etc.)
  agents/                  ← agent definitions (generated by /bootstrap for your project)

tasks/
  available/               ← TODO + all dependencies DONE
  in-progress/             ← currently being worked on
  ready-for-pr/            ← implemented, waiting for /prepare-pr
  pr-open/                 ← PR open on GitHub
  done/                    ← merged
  blocked/                 ← dependencies not yet DONE

context/
  decisions.md             ← log of technical decisions (agents write here)
  discoveries.md           ← cross-agent alerts (agents write here)

docs/                      ← project documentation (API, ADRs, architecture)
```

---

## Task lifecycle

```
available/ → in-progress/ → ready-for-pr/ → pr-open/ → done/
```

Status lives in the task file's YAML frontmatter. Moving between folders is the visual signal — frontmatter is the machine-readable source of truth.

### Task file format

```markdown
---
id: T-001
phase: 0
agent: backend
depends_on: []
status: available
folders: [apps/api/, libs/auth/]
outputs: [POST /auth/login, GET /auth/token]
size: M
branch: ~
pr: ~
---

## Title

**Scope:** what modules this touches.

**Delivers:**
- specific function or endpoint
- another deliverable

**Done when:**
- [ ] acceptance criterion 1
- [ ] acceptance criterion 2
- [ ] tests written and passing
- [ ] docs/api.md updated (if new endpoint)
```

Max 150 words in the body. Checklists, not prose.

After completing a task, the agent appends:

```markdown
## Completed
- What was implemented
- What changed from the original plan
- Decisions made and why
```

## Agent ownership

Each agent owns specific folders. It **never writes outside them**.

Folder ownership is defined in each agent's file in `.claude/agents/`. When `/bootstrap` generates agents for your project, it assigns folders based on your architecture.

**Protected files** — require explicit human approval before any agent touches them:
- Shared contracts (e.g., `libs/common/models.py` or equivalent)
- Database schema files
- CI/CD configuration

Any change to protected files must be proposed, approved, and updated in all consuming modules in the same PR.

---

## Branch policy

```
main                          ← always stable, all checks passing
feature/T-XXX-short-slug      ← one branch per task
```

- Never commit directly to `main` (except `tasks/` status updates)
- Each agent works in its own git worktree (`../project-T-XXX/`)
- Branch creation = task claim. If push fails, another agent claimed it first → pick another task
- `tasks/*.md` status updates (IN_PROGRESS, READY_FOR_PR, DONE) are pushed directly to main — they are coordination metadata, not production code

> **Requirement:** `main` must **not** be a protected branch. The framework pushes `tasks/*.md` status metadata directly to `main` as its coordination mechanism (`/orchestrate`, `/done`, `/add-task`, `/bug`). Branch protection on `main` will make these pushes fail and break the workflow. If your org requires protection, exclude `tasks/**` from the ruleset — otherwise leave `main` unprotected.

## Quality gates

Every PR must pass before merge:

**Tests**
- All tests pass
- Coverage on core modules ≥ threshold in `devteam.config.yml` (default 70%)
- Unit tests written by implementing agent in same PR
- No network calls in unit tests

**Code quality**
- Linting passes (tool configured per stack in `devteam.config.yml`)
- Formatting clean
- Type checking passes
- No secrets, debug prints, or commented-out code

**Architecture**
- No imports outside assigned folders
- No business logic in HTTP layer
- Protected files untouched (or explicitly approved)

**PR Review sub-agents** (run by PR Reviewer automatically):
- Code Quality Agent — scope, patterns, correctness
- Adversarial Agent — activates on unanimous approval, actively looks for flaws
- Security Agent — OWASP top 10 on the diff
- Smoke Test Agent — spins up app, tests acceptance criteria against real/sandbox API
- Mutation Test Agent — for critical modules only, verifies tests catch real bugs

**Documentation**
- New endpoint → update API docs
- Architecture decision → create ADR in `docs/adr/`
- Setup change → update README

## Context files

### `context/decisions.md`
Log of technical decisions. Any agent that makes a non-obvious choice writes here:
```
## YYYY-MM-DD — T-XXX [Agent name]
Decided: [what]
Why: [reason]
Affects: [files/modules]
Discarded: [alternative and why not]
```

### `context/discoveries.md`
Cross-agent alerts. When an agent finds something that affects another module:
```
## OPEN — YYYY-MM-DD [Source agent → Target agent]
[What was found and what action is needed]
Status: open / resolved in T-XXX
```

The Orchestrator reads both files at the start of each task and surfaces relevant entries in the human checkpoint.

## Available commands

| Command | What it does |
|---------|-------------|
| `/team-init` | Configure the project and show current state — run this first |
| `/bootstrap` | Conversational setup: idea → design → plan → tasks → infra |
| `/orchestrate` | Pick next task, implement, push, mark READY_FOR_PR |
| `/prepare-pr T-XXX` | Review, run all checks, open PR |
| `/done T-XXX` | Mark DONE after merge, report unblocked tasks |
| `/status` | Full project status board |
| `/explore [topic]` | Design exploration before coding |
| `/add-task [description]` | Design and add a new task mid-project |
| `/guide` | Current state of the project — what's built, how to test it |
| `/bug [description]` | Investigate and fix a bug |
| `/restart T-XXX` | Recover a task stuck in-progress (agent crashed, worktree lost) |
| `/cancel T-XXX` | Abandon a task cleanly — removes from active flow, preserves audit trail |

## Rules agents must follow

1. **Never touch files outside assigned folders** — even if you see an improvement
2. **Never modify shared contracts** without explicit human approval
3. **Always write tests** in the same PR as the implementation
4. **Always update `context/decisions.md`** when making non-obvious choices
5. **Always check `context/discoveries.md`** before starting implementation
6. **Never push code directly to main** — only `tasks/*.md` status updates are the exception
7. **Never skip the human checkpoint** — present plan, wait for confirmation, then code
8. **Always update the task** with what was done after marking READY_FOR_PR
