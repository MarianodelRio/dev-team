# dev-team

**Framework for building software with a team of AI agents via parallel spec-driven development.**

From a vague idea to production code — specialized agents working in parallel, each owning their module, enforcing quality at every step.

---

## How it works

```
Your idea  →  design session  →  tasks  →  parallel agents  →  reviewed PRs  →  shipped code
```

1. Write your idea in `IDEA.md` (no technical knowledge required)
2. Run `/bootstrap` — collaborative design conversation that generates architecture, plan, and tasks
3. Run `/orchestrate` — an agent picks the best available task, presents a plan, waits for your approval, then implements
4. Run `/prepare-pr T-XXX` — full review with specialized sub-agents (security, smoke tests, adversarial review)
5. Merge on GitHub, run `/done T-XXX` — marks done, tells you what's now unblocked

Multiple agents can work in parallel on different tasks using git worktrees — no conflicts, no coordination overhead.

## Quick start

> **Requirement:** keep `main` **unprotected**. dev-team pushes task status metadata (`tasks/*.md`) directly to `main` as its coordination mechanism — branch protection will break the workflow. If your org mandates protection, exclude `tasks/**` from the ruleset.

### New project

1. Click **"Use this template"** on GitHub to create your repo
2. Clone it locally and open it in Claude Code
3. Run `/team-init` — configure your project, set your stack, describe your idea
4. Run `/bootstrap` — design conversation → architecture + tasks generated
5. Run `/orchestrate` — agents start implementing

### Existing project

```bash
# Copy the framework files into your project
curl -fsSL https://raw.githubusercontent.com/MarianodelRio/dev-team/main/install.sh | bash

# Then open Claude Code and run:
# /team-init          ← configure the framework for this project
# /bootstrap --mode=brownfield  ← analyzes your codebase, generates delta tasks
```

## What gets generated by `/bootstrap`

- `design.md` — architecture, modules, tech stack, contracts
- `plan.md` — phases and dependency graph
- `tasks/` — all tasks with acceptance criteria
- `CLAUDE.md` — project-specific rules (this file becomes project-specific)
- `.claude/agents/` — specialized agents for your exact modules and stack
- `docker-compose.yml`, `Dockerfile` — if your project needs it
- `.github/workflows/ci.yml` — CI pipeline
- `.env.example`, `.gitignore`, `CONTRIBUTING.md`, `LICENSE`, `SECURITY.md`

## Entry points

dev-team adapts to where you are:

| You have | Run |
|----------|-----|
| A vague idea | `/bootstrap` → starts with ideation conversation |
| A complete idea | `/bootstrap` → starts at design |
| A design already | `/bootstrap` → starts at planning |
| Tasks defined | `/bootstrap` → validates and generates agents + infra |
| Existing codebase | `/bootstrap --mode=brownfield` → analyzes, generates delta tasks |

## Key features

**Parallel execution** — Multiple agents work simultaneously in isolated git worktrees. No conflicts. Each owns specific folders and cannot touch others.

**Spec-first** — Design and plan are approved before any code is written. The human checkpoint is mandatory and cannot be skipped.

**Quality by default** — Every PR goes through Code Quality, Security (OWASP), Smoke Tests against real APIs, and an Adversarial reviewer that activates on unanimous approval.

**Batteries included** — Docker, CI, .env, .gitignore, security policy, PR templates — all generated automatically for your stack.

**Model-agnostic configuration** — Configure which model handles reasoning vs. implementation vs. fast tasks in `devteam.config.yml`.

**Adaptive to your level** — Bootstrap conversation adapts to your technical expertise. Works for senior engineers and non-technical founders alike.

## Commands

```
/team-init          Configure the project and show current state — start here
/bootstrap          Start or resume the design conversation
/orchestrate        Pick and implement the next available task
/prepare-pr T-XXX   Full review and open PR
/done T-XXX         Mark task done after merge, clean up branch
/status             Full project status board
/add-task           Design and add a new task mid-project
/guide              Current state — what's built, how to test it
/cheatsheet         What to do next, contextual to the board (global or /cheatsheet T-XXX)
/explore [topic]    Design exploration before coding
/bug [description]  Investigate and fix a bug
/restart T-XXX      Recover a task stuck in-progress (agent crashed)
/cancel T-XXX       Abandon a task cleanly
```

## Configuration

`devteam.config.yml` controls all framework behavior: model selection, PR mode (automatic/manual), quality gate thresholds, smoke test mode, and which batteries to include.

See [devteam.config.yml](devteam.config.yml) for all options with documentation.

## Why dev-team?

Most AI coding tools are single-agent: one agent, one context, one task at a time. dev-team is different:

- **Specialized agents** know their module deeply — the ML agent understands your ML stack, the frontend agent understands your UI framework
- **Parallel execution** — real parallelism via git worktrees, not simulated
- **Folder ownership** — agents cannot touch what isn't theirs, preventing the coordination failures that break multi-agent systems
- **Mandatory human checkpoints** — you approve the plan before code is written, every time
- **Quality is non-negotiable** — security, smoke tests, adversarial review, mutation testing — all automatic
