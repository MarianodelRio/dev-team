# dev-team

**Framework for building software with a team of AI agents via parallel spec-driven development.**

From a vague idea to production code ‚Äî specialized agents working in parallel, each owning their module, enforcing quality at every step.

---

## How it works

```
Your idea  ‚Üí  design session  ‚Üí  tasks  ‚Üí  parallel agents  ‚Üí  reviewed PRs  ‚Üí  shipped code
```

1. Write your idea in `IDEA.md` (no technical knowledge required)
2. Run `/bootstrap` ‚Äî collaborative design conversation that generates architecture, plan, and tasks
3. Run `/orchestrate` ‚Äî an agent picks the best available task, presents a plan, waits for your approval, then implements
4. Run `/prepare-pr T-XXX` ‚Äî full review with specialized sub-agents (security, smoke tests, adversarial review)
5. Merge on GitHub, run `/done T-XXX` ‚Äî marks done, tells you what's now unblocked

Multiple agents can work in parallel on different tasks using git worktrees ‚Äî no conflicts, no coordination overhead.

## Quick start

### New project

1. Click **"Use this template"** on GitHub to create your repo
2. Clone it locally and open it in Claude Code
3. Run `/team-init` ‚Äî configure your project, set your stack, describe your idea
4. Run `/bootstrap` ‚Äî design conversation ‚Üí architecture + tasks generated
5. Run `/orchestrate` ‚Äî agents start implementing

### Existing project

```bash
# Copy the framework files into your project
curl -fsSL https://raw.githubusercontent.com/MarianodelRio/dev-team/main/install.sh | bash

# Then open Claude Code and run:
# /team-init          ‚Üê configure the framework for this project
# /bootstrap --mode=brownfield  ‚Üê analyzes your codebase, generates delta tasks
```

## What gets generated by `/bootstrap`

- `design.md` ‚Äî architecture, modules, tech stack, contracts
- `plan.md` ‚Äî phases and dependency graph
- `tasks/` ‚Äî all tasks with acceptance criteria
- `CLAUDE.md` ‚Äî project-specific rules (this file becomes project-specific)
- `.claude/agents/` ‚Äî specialized agents for your exact modules and stack
- `docker-compose.yml`, `Dockerfile` ‚Äî if your project needs it
- `.github/workflows/ci.yml` ‚Äî CI pipeline
- `.env.example`, `.gitignore`, `CONTRIBUTING.md`, `LICENSE`, `SECURITY.md`

## Key features

**Parallel execution** ‚Äî Multiple agents work simultaneously in isolated git worktrees. No conflicts. Each owns specific folders and cannot touch others.

**Spec-first** ‚Äî Design and plan are approved before any code is written. The human checkpoint is mandatory and cannot be skipped.

**Quality by default** ‚Äî Every PR goes through Code Quality, Security (OWASP), Smoke Tests against real APIs, and an Adversarial reviewer that activates on unanimous approval.

**Batteries included** ‚Äî Docker, CI, .env, .gitignore, security policy, PR templates ‚Äî all generated automatically for your stack.

**Model-agnostic configuration** ‚Äî Configure which model handles reasoning vs. implementation vs. fast tasks in `devteam.config.yml`.

**Adaptive to your level** ‚Äî Bootstrap conversation adapts to your technical expertise. Works for senior engineers and non-technical founders alike.

## Commands

```
/team-init          Configure the project and show current state ‚Äî start here
/bootstrap          Start or resume the design conversation
/orchestrate        Pick and implement the next available task
/prepare-pr T-XXX   Full review and open PR
/done T-XXX         Mark task done after merge, clean up branch
/status             Full project status board
/add-task           Design and add a new task mid-project
/guide              Current state ‚Äî what's built, how to test it
/explore [topic]    Design exploration before coding
/bug [description]  Investigate and fix a bug
/restart T-XXX      Recover a task stuck in-progress (agent crashed)
/cancel T-XXX       Abandon a task cleanly
```

## Why dev-team?

Most AI coding tools are single-agent: one agent, one context, one task at a time. dev-team is different:

- **Specialized agents** know their module deeply ‚Äî the ML agent understands your ML stack, the frontend agent understands your UI framework
- **Parallel execution** ‚Äî real parallelism via git worktrees, not simulated
- **Folder ownership** ‚Äî agents cannot touch what isn't theirs, preventing the coordination failures that break multi-agent systems
- **Mandatory human checkpoints** ‚Äî you approve the plan before code is written, every time
- **Quality is non-negotiable** ‚Äî security, smoke tests, adversarial review, mutation testing ‚Äî all automatic
